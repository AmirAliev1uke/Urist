# План: миграция на Qdrant с авто-классификацией документов

## Ключевые решения (по вашим ответам)

1. **pgvector полностью убирается** как векторная БД. Postgres остаётся только для метаданных (`knowledge_documents`, `analyses`, будущие `users`)
2. **Qdrant хранит всё**: и судебную практику, и законы. Векторы + тексты чанков + payload
3. **Авто-классификация** при загрузке — без ручного ввода:
   - **Год решения** — регулярные выражения (базовые, расширяемые позже)
   - **Тип суда** — маркеры в тексте + паттерны номеров дел (А40... = арбитраж)
   - **doc_type** — определяется по контенту: закон vs судебное решение
4. Для законов поля `year` и `court_level` = null

## Авто-классификация — новый модуль `app/core/classifier.py`

Определяет метаданные из текста документа БЕЗ ИИ, регулярками и словарями.

**`extract_metadata(text, file_name) -> DocumentMetadata`:**
```python
@dataclass
class DocumentMetadata:
    doc_type: str          # "law" | "judicial_practice"
    year: int | None       # год решения (для law = None)
    court_level: str | None  # "arbitration" | "general" (для law = None)
    case_number: str | None  # "А40-12345/2021" если найден
```

**Логика определения:**
- `doc_type` = "judicial_practice" если есть маркеры судебного решения (постановление/решение/определение + номер дела), иначе "law"
- `court_level` = "arbitration" если найден паттерн номера `А\d+-\d+/\d+` или слова «Арбитражный суд»/«АС », иначе "general"
- `year` = ищем «от DD.MM.YYYY», «от DD месяца YYYY», «N.../YYYY»

Паттерны вынесены в константы вверху файла — легко расширять.

## Payload-схема чанка в Qdrant

Каждый чанк (point) получает:
```python
{
    "id": "<uuid>",
    "vector": [0.03, -0.01, ...],  # 384-dim
    "payload": {
        "document_id": 42,           # связь с knowledge_documents в Postgres
        "text": "Статья 779. ...",
        "article_ref": "ст. 779",     # для законов
        "doc_type": "judicial_practice",
        "year": 2021,                # для law = None
        "court_level": "arbitration", # для law = None
        "case_number": "А40-12345/2021",
        "source_title": "Постановление 9 ААС от...",
        "chunk_index": 0
    }
}
```

## Этапы реализации

### Этап 1 — Инфраструктура Qdrant
- `docker-compose.yml`: сервис `qdrant` (образ `qdrant/qdrant`, порт 6333, volume `qdrant_data`)
- `requirements.txt`: `qdrant-client>=1.12.0` вместо `pgvector`
- `.env`: `QDRANT_URL=http://qdrant:6333`, `QDRANT_API_KEY=` (для cloud позже), `QDRANT_COLLECTION=legal_chunks`
- `config.py`: настройки Qdrant

### Этап 2 — Модуль авто-классификации `classifier.py`
- `extract_metadata(text)` → `DocumentMetadata`
- Регулярки для даты (3-4 паттерна под российский формат)
- Маркеры суда (словарь + regex для номеров дел)
- Базовая реализация, легко расширяется

### Этап 3 — Переписать `vector_store.py` на Qdrant
Сигнатуры публичных функций сохраняются (чтобы не трогать LLM-клиенты), внутренности меняются:
- `add_chunks()` — `qdrant_client.upsert()` с payload (включая year, court_level из метаданных)
- `similarity_search()` — `qdrant_client.query_points()`, преобразование в `SearchResult`
- `delete_document()` — `qdrant_client.delete(filter=document_id)`
- Создание коллекции при старте (квантизация + payload-индексы)
- `SearchResult` — без изменений (контракт для LLM-клиентов)

### Этап 4 — Очистить Postgres от векторов
- `models.py`: удалить `DocumentChunk`, убрать `Vector`, убрать relationship `chunks`
- `init.sql`: убрать `CREATE EXTENSION vector`, таблицу `document_chunks`, HNSW-индекс
- Очистить volume pgdata (данные тестовые)

### Этап 5 — RAG-оркестрация
- `rag.py`: `ingest_document` вызывает `classifier.extract_metadata()` первым делом, передаёт метаданные в `add_chunks`
- Qdrant-клиент инжектится через зависимости (синглтон)

### Этап 6 — Health-check и API
- `health.py`: `/health/db` теперь проверяет и Postgres, и Qdrant (count points)
- `knowledge.py`: без изменений в сигнатуре (метаданные определяются автоматически)

### Этап 7 — Запуск и проверка
- `docker compose down -v` (очистка старых томов pgvector)
- `docker compose up --build`
- Тесты:
  1. Загрузить закон → проверить doc_type=law, year=null
  2. Загрузить судебное решение с датой → проверить year, court_level
  3. Анализ документа → search через Qdrant
  4. Фильтр по году через Python-проверку

## Что НЕ делаем в этой итерации
- Перенос данных (старые тестовые данные сносим)
- Гибридный поиск (BM25 + векторы) — Qdrant поддерживает, добавим если понадобится
- UI-фильтры для анализа (по году/суду) — позже
- Кластер Qdrant (один узел для начала)
- Дообучение регулярок классификатора (базовая версия, расширится на реальных данных)
