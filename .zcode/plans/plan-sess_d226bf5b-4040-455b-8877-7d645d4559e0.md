# План: AI-Ассистент для юристов (Legal AI Assistant)

## Архитектура RAG-системы (Retrieval-Augmented Generation)

```
┌─────────────┐     ┌──────────────────────────┐     ┌──────────────┐
│   React     │────▶│      FastAPI (Python)    │────▶│  PostgreSQL  │
│  Frontend   │◀────│                          │◀────│  + pgvector  │
└─────────────┘     │  ┌────────────────────┐  │     └──────────────┘
                    │  │ LLM Layer (stub)   │  │
                    │  │ + GigaChat/OpenAI  │  │     ┌──────────────┐
                    │  └────────────────────┘  │────▶│  HuggingFace │
                    │  ┌────────────────────┐  │     │  embeddings  │
                    │  │  PDF Parser        │  │     │  (локально)  │
                    │  └────────────────────┘  │     └──────────────┘
                    │  ┌────────────────────┐  │
                    │  │  RAG Orchestration │  │
                    │  └────────────────────┘  │
                    └──────────────────────────┘
```

## Два разных потока данных (важно различать)

**Поток A — Наполнение базы знаний (загружаете вы):**
PDF (ГК РФ, НК РФ, судебная практика) → извлечение текста → чанкование по статьям/абзацам → embedding (локально) → сохранение в pgvector.

**Поток B — Анализ документа юриста:**
Загрузка PDF → извлечение текста → embedding запроса → поиск релевантных статей/практики в pgvector (top-K) → сборка промпта (документ + найденный контекст) → отправка в LLM → структурированный ответ с рекомендациями и подсветкой.

## Этапы реализации

### Этап 1 — Инфраструктура и окружение
- Инициализация git-репозитория
- `docker-compose.yml`: 3 сервиса — `db` (pgvector/pgvector:pg16), `backend`, `frontend`
- `.env.example` с переменными (`DATABASE_URL`, `LLM_PROVIDER=stub`, и т.д.)
- Dockerfiles для backend (Python 3.12) и frontend (Node 22 + nginx)

### Этап 2 — База данных и векторное хранилище
- Модели таблиц:
  - `knowledge_documents` (id, title, doc_type, source_url, created_at)
  - `document_chunks` (id, document_id, chunk_index, text, embedding vector(384))
  - `analyses` (id, filename, status, result_json, created_at)
- SQL-миграции: создание расширения `vector`, индекс HNSW/IVFFlat для поиска
- `vector_store.py`: `add_documents()`, `similarity_search(query_embedding, top_k)`

### Этап 3 — PDF-парсинг и чанкование
- `pdf_parser.py` — PyMuPDF (быстрый, хорошее качество для русского текста)
- `chunker.py` — семантическое чанкование: разбиение по статьям/пунктам (юридические документы имеют иерархию), fallback на чанки с перекрытием
- Сохранение метаданных (номер статьи, источник) для цитирования в ответе

### Этап 4 — Embeddings (локально, бесплатно)
- `embeddings.py` — модель `sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2` (384 измерения, ~118 МБ, отличная работа с русским)
- Предзагрузка модели в Dockerfile при сборке образа
- Методы: `embed_text(str) -> list[float]`, `embed_batch(list[str]) -> list[list[float]]`

### Этап 5 — Слой абстракции LLM + Stub
- `BaseLLMClient` (ABC) с методом `analyze(document_text, retrieved_context) -> AnalysisResult`
- `StubLLMClient` — возвращает реалистичный мок-ответ (рекомендации, риски, цитаты) — позволяет разрабатывать и тестировать фронтенд БЕЗ реального API
- `factory.py` — выбор по `LLM_PROVIDER` из .env (`stub` / `gigachat` / `openai`)
- Заготовки классов `gigachat.py`, `openai.py` — подключите позже, подставив свой API-ключ

### Этап 6 — RAG-оркестрация (сердце системы)
- `rag.py` — связывает всё:
  1. Извлечение текста из загруженного документа
  2. Извлечение ключевых фраз → embedding → поиск top-K релевантных норм в pgvector
  3. Сборка промпта с системной инструкцией: «Ты — юридический ассистент. Опирайся только на предоставленные нормы права и судебную практику. Указывай ссылки на статьи.»
  4. Вызов LLM (через абстракцию)
  5. Парсинг ответа в структуру: рекомендации, риски, подсветка

### Этап 7 — REST API
- `POST /api/knowledge/upload` — загрузка PDF в базу знаний (Поток A)
- `GET /api/knowledge/documents` — список загруженных источников
- `DELETE /api/knowledge/documents/{id}` — удаление
- `POST /api/analyze` — загрузка документа для анализа (Поток B), возвращает `analysis_id`
- `GET /api/analyze/{id}` — получение результата анализа
- CORS настроен для фронта

### Этап 8 — React фронтенд
- Vite + React + TypeScript + TanStack Query + react-dropzone
- **Главный экран:** drag&drop загрузки документа для анализа
- **Экран результата:** 
  - Левая панель — документ с подсветкой (риски — красным, рекомендации — жёлтым, ссылки на статьи — синим)
  - Правая панель — структурированный отчёт ИИ: рекомендации, найденные риски, релевантные статьи с цитатами
- **Админ-панель базы знаний:** загрузка PDF-источников, просмотр списка
- Подсветка реализована через match цитат из ответа LLM по тексту документа

### Этап 9 — Документация и запуск
- `README.md` с инструкциями: запуск `docker compose up`, как загрузить базу знаний, как переключить LLM-провайдер
- `.env.example` с комментариями

## Технологии

| Компонент | Технология | Причина |
|-----------|-----------|---------|
| Backend | FastAPI (Python 3.12) | Async, быстрый, автодокументация |
| БД | PostgreSQL 16 + pgvector | Реляционные данные + векторы в одной БД |
| Embeddings | sentence-transformers (MiniLM multilingual) | Бесплатно, локально, хорошо с русским |
| PDF парсинг | PyMuPDF (fitz) | Быстрый, качественный |
| Frontend | React 18 + Vite + TypeScript | По вашему выбору |
| Контейнеризация | Docker Compose | По вашему выбору |

## Что НЕ входит в MVP (отмечено на будущее)
- Аутентификация пользователей (single-workspace для начала)
- Очереди задач (Celery) для длинных анализов — на MVP синхронно с лёгкой асинхронностью
- Экспорт отчётов в PDF/DOCX
- История анализов с поиском

## С чего начнём реализацию
Этапы 1–5 сначала (инфраструктура + БД + парсинг + embeddings + stub LLM), чтобы получить работающий скелет. Затем Этап 6–7 (RAG + API). Фронтенд (Этап 8) можно делать параллельно со stub-ответами бэкенда.
